# MCP Knowledge (Database Knowledge Graph (DBKG))

This page is the database-backed MCP knowledge model for ConvEngine. It documents the new `dbkg.*` tool chain, the `ce_*` metadata package behind it, and the runtime safety model that makes the DB investigation flow configurable instead of hardcoded in Java.

<Highlight type="info" title="Core idea">
`dbkg.*` is not a raw SQL generator. It is a DB-driven investigation pipeline: resolve the case, load graph knowledge, choose a playbook, validate the playbook graph, then execute approved query steps.
</Highlight>


## Story first: how Database Knowledge Graph (DBKG) works from a real user question

Start from the operator, not the table. A real user does not know `zp_ui_data_history`, `ce_mcp_query_template`, or `ce_mcp_playbook_transition`. They speak in support language, and the engine has to do the translation. The story below is intentionally broken into real turn-by-turn cards so you can see exactly what happens before, during, and after the Database Knowledge Graph (DBKG) tool chain.

    throw new IllegalArgumentException(source + " SQL must begin with SELECT or WITH for read-only execution.");
  }
  if (SELECT_INTO.matcher(normalized).find()) {
    throw new IllegalArgumentException(source + " SQL uses SELECT INTO, which is not allowed in read-only mode.");
  }
  if (FOR_UPDATE.matcher(normalized).find()) {
    throw new IllegalArgumentException(source + " SQL uses FOR UPDATE, which is not allowed in read-only mode.");
  }
  ...
  if (!rules.allowedFunctions.isEmpty()) {
    ...
  }
}`}
</CodeBlockToggle>

What it does:
- blocks write/admin statements such as `INSERT`, `UPDATE`, `DELETE`, `DROP`, `TRUNCATE`, `ALTER`, `CREATE`, `MERGE`, and `CALL`
- rejects multi-statement SQL
- rejects `SELECT INTO` and `FOR UPDATE`
- loads extra rules from `ce_mcp_sql_guardrail`
- optionally enforces a function allowlist when `ALLOW_FUNCTION` rows exist

Why it matters: read-only is enforced in Java now, not just implied by convention or metadata.

### SQL trace observability

Both SQL-backed MCP paths now emit dedicated audit stages and optional `ce_verbose` hooks:
- `MCP_DB_SQL_EXECUTION` from `McpDbExecutor` for classic `ce_mcp_db_tool` SQL templates
- `DBKG_QUERY_SQL_EXECUTION` from `DbkgQueryTemplateStepExecutor` for Database Knowledge Graph (DBKG) query-template steps

Both payloads include:
- `sql`
- `params`
- `row_count`
- `rows`

That gives consumers one place to inspect the executed SQL text and the returned row set, and one place to override the user-facing progress text through `ce_verbose`.
