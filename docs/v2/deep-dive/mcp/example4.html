MCP Example 4 - Zapper Disconnect

This is the concrete Database Knowledge Graph (DBKG) example. It uses the new transaction schema under src/main/resources/sql/zapper plus the Database Knowledge Graph (DBKG) metadata package under src/main/resources/sql/mcp/dbkg.

The example shows four production-style questions against real tables:

  • assigned then rejected in the last 24 hours
  • inventory not found during ASR handling
  • submitted disconnect older than 24 hours without final disconnect id
  • disconnect id exists in order but is missing in BillBank
Important runtime expectation

The cleanest way to test this flow is with the Database Knowledge Graph (DBKG) planner scope (intent=DBKG_DIAGNOSTICS, state=ANALYZE) so the seeded planner prompt forces validate -> execute ordering.

How this trace maps to real runtime behavior

Each user turn below now shows exactly one real assistant message. The same assistant bubble includes the internal MCP trace in its JSON/details section so the example stays faithful to actual ConvEngine runtime behavior while still showing what happened inside the Database Knowledge Graph (DBKG) tool chain.

Audit trail you should expect

StageWhat is writtenWhy it matters
MCP_TOOL_CALLEvery dbkg.* tool invocation and args.Lets you confirm the planner chose the correct tool order.
MCP_TOOL_RESULTSerialized JSON result for each Database Knowledge Graph (DBKG) tool.This is how context.mcp.observations is built and how the next planner turn sees prior evidence.
MCP_TOOL_ERRORValidation or guardrail failure reasons such as DBKG_VALIDATION_REQUIRED_BEFORE_EXECUTION.Makes skipped execution explainable in production.
POST_AGENT_MCPRule reactions after Database Knowledge Graph (DBKG) finishes.Consumers can still derive state transitions or final response behavior after the tool chain completes.

Database Knowledge Graph (DBKG) query templates used by this example
SQL
-- Transition audit
SELECT ...
FROM zp_ui_data_history h1
JOIN zp_ui_data_history h2 ON ...
WHERE h1.zp_action_id = :from_action_id
AND h2.zp_action_id = :to_action_id
AND h2.created_date >= :from_ts;

-- Inventory linkage
SELECT ...
FROM zp_inventory_service i
WHERE (:account_id IS NULL OR i.account_id = :account_id)
AND (:acc_loc_id IS NULL OR i.acc_loc_id = :acc_loc_id)
AND (:zp_connection_id IS NULL OR i.zp_connection_id = :zp_connection_id);

-- Disconnect chain
SELECT ...
FROM zp_disconnect_order o
WHERE o.submitted_at <= :from_ts
AND (o.zp_disconnect_id IS NULL OR UPPER(o.downstream_status) <> 'COMPLETED');

-- BillBank reconciliation
SELECT ...
FROM zp_disconnect_order o
LEFT JOIN zp_billbank_record b ON o.zp_disconnect_id = b.zp_disconnect_id
WHERE o.zp_disconnect_id IS NOT NULL
AND b.billbank_id IS NULL;

Why the engine gives the answer it gives​

How Database Knowledge Graph (DBKG) turns rows into explanations

LayerWhat Database Knowledge Graph (DBKG) seesWhat the operator gets
Case resolutionWeighted signal match against business wording.The right investigation family is selected without the operator naming a table.
Knowledge lookupSystems, entities, API flows, statuses, live schema objects.The planner gets context about where to look and which IDs matter.
Playbook planA concrete DAG of steps and transitions.Execution order becomes consistent and auditable.
ValidationGraph integrity check before SQL.Broken metadata is blocked before it can produce misleading answers.
ExecutionApproved templates + bound params + step outputs.Evidence comes from known-safe queries instead of ad hoc SQL.
OutcomeCondition-driven diagnosis.Support teams get an explanation tied to evidence, not just a dump of rows.
