# MCP Semantic Catalog (`db.semantic.catalog`)

This page documents the semantic DB metadata lookup path used by MCP when tool code is `db.semantic.catalog`.

<Highlight type="info" title="What this tool is">
`db.semantic.catalog` is a ranking and grounding tool. It returns schema/query hints from metadata tables. It does not execute business SQL.
</Highlight>

## Where it fits

`db.semantic.catalog` sits between free-form user language and executable SQL tools such as `postgres.query`.

1. Planner receives user request and current MCP observations.
2. Planner calls `db.semantic.catalog` with a text question.
3. Semantic catalog returns ranked query + schema knowledge.
4. Planner uses those hints to decide next tool call or answer.
5. If SQL is needed, planner calls `postgres.query` separately.

## Ranking model

`DbSemanticCatalogService` ranking behavior is deterministic:

1. tokenize user question with lowercase normalization
2. split by non-alphanumeric separators
3. remove stop words and tiny tokens
4. build candidate document tokens from row text fields
5. compute overlap-based score using harmonic mean of precision + recall
6. filter by `min-score`
7. return top `max-results`

<Highlight type="tip" title="Practical implication">
Quality is mostly driven by good `tags`, `description`, and realistic `prepared_sql` examples in knowledge tables.
</Highlight>

## Capsule and ranking behavior

When `knowledge-capsule=true`, semantic catalog returns `dbkgCapsule` directly from semantic metadata tables.

- Planner consumes this capsule as primary grounding context (`{{dbkg_capsule}}`).
- This avoids relying only on tiny ranked `schemaKnowledge` subsets when your semantic catalog is large.
- Ranked lists still exist (`queryKnowledge`, `schemaKnowledge`) for next-step guidance.

If `vector-search.enabled=true`, ranking can run via interceptor path:

- framework default: embedding + Java cosine fallback
- consumer override: DB-native vector search (for example `pgvector` cosine distance)

## Config surface

```yaml
convengine:
  mcp:
    db:
      semantic-catalog:
        enabled: true
        knowledge-capsule: true
        schema-knowledge: true
        query-knowledge: true
        max-results: 5
        scan-limit: 5000
        min-score: 0.15
        query-catalog-table: ce_mcp_query_knowledge
        query-text-column: query_text
        query-description-column: description
        prepared-sql-column: prepared_sql
        tags-column: tags
        api-hints-column: api_hints
        schema-catalog-table: ce_mcp_schema_knowledge
        schema-table-name-column: table_name
        schema-column-name-column: column_name
        schema-valid-values-column: valid_values
        schema-description-column: description
        schema-tags-column: tags
        vector-search:
          enabled: false
          max-results: 10
          vector-column: vector
```

<Highlight type="info" title="No semantic tool-code property">
`db.semantic.catalog` is fixed in the handler and registered through `ce_mcp_tool` rows. There is no `semantic-catalog.tool-code` property.
</Highlight>

<Highlight type="tip" title="Selective source toggles">
When `query-knowledge=false`, the runtime skips `ce_mcp_query_knowledge`. When `schema-knowledge=false`, it skips `ce_mcp_schema_knowledge`. When `knowledge-capsule=false`, planner capsule extraction/build is skipped.
</Highlight>

## Registration and seeding

`ce_mcp_tool` must contain a row for semantic catalog:

```sql
INSERT INTO ce_mcp_tool (tool_code, tool_group, intent_code, state_code, description, enabled)
VALUES ('db.semantic.catalog', 'DB', 'ANY', 'ANY', 'Semantic query/schema knowledge retrieval', true);
```

`ce_mcp_db_tool` is not required for this tool because execution is handler-based (`DbSemanticCatalogToolHandler`), not SQL-template based.

Recommended `ce_mcp_schema_knowledge` shape:

```sql
CREATE TABLE IF NOT EXISTS ce_mcp_schema_knowledge (
  id BIGINT PRIMARY KEY,
  table_name VARCHAR(255) NOT NULL,
  column_name VARCHAR(255),
  valid_values VARCHAR(2000),
  vector VECTOR, -- consumer-managed (or TEXT if you store serialized vectors)
  description VARCHAR(2000),
  tags VARCHAR(1000)
);
```

## Consumer vector interceptor example

```java
@Component
@Order(10) // higher priority than framework fallback
@RequiredArgsConstructor
public class PgVectorSemanticSearchInterceptor implements SemanticCatalogVectorSearchInterceptor {

  private final JdbcTemplate jdbcTemplate;
  private final LlmClient llmClient;

  @Override
  public boolean supports(ConvEngineMcpConfig.Db.Knowledge.VectorSearch config) {
    return config != null && config.isEnabled();
  }

  @Override
  public List<Map<String, Object>> rank(
      String sourceType,
      String question,
      List<Map<String, Object>> rows,
      ConvEngineMcpConfig.Db.Knowledge cfg) {

    float[] embedding = llmClient.generateEmbedding(question);
    String embeddingLiteral = toPgVectorLiteral(embedding);

    String table = "query".equalsIgnoreCase(sourceType)
        ? cfg.getQueryCatalogTable()
        : cfg.getSchemaCatalogTable();

    int maxResults = Math.max(1, cfg.getVectorSearch().getMaxResults());
    String vectorCol = cfg.getVectorSearch().getVectorColumn();

    String sql = """
      SELECT t.*, (1 - (t.%s <=> ?::vector)) AS _vector_score
      FROM %s t
      WHERE t.%s IS NOT NULL
      ORDER BY t.%s <=> ?::vector
      LIMIT ?
      """.formatted(vectorCol, table, vectorCol, vectorCol);

    return jdbcTemplate.queryForList(sql, embeddingLiteral, embeddingLiteral, maxResults);
  }

  private String toPgVectorLiteral(float[] vector) {
    StringBuilder sb = new StringBuilder("[");
    for (int i = 0; i < vector.length; i++) {
      if (i > 0) sb.append(",");
      sb.append(vector[i]);
    }
    sb.append("]");
    return sb.toString();
  }
}
```

## Postgres query interceptor (execution safety)

`postgres.query` now supports consumer-overridable SQL interceptors before read-only guardrail + execution.

Runtime shape:

1. Planner generates SQL.
2. `PostgresQueryToolHandler` runs interceptor chain (`PostgresQueryInterceptor` beans by order).
3. Normalized SQL is validated by `McpSqlGuardrail`.
4. Query executes.

Framework default interceptor (`DefaultPostgresQueryInterceptor`, lowest precedence) auto-fixes a common LLM error:

- bad: `to_timestamp(ts_col/1000.0)` when `ts_col` is already `timestamp/date`
- fix: rewrites to raw `ts_col`

Column type is resolved from `information_schema.columns` under current schemas, so this is schema-aware and avoids runtime errors like:

- `operator does not exist: timestamp with time zone / numeric`

Consumer override example:

```java
@Component
@Order(10) // higher priority than framework fallback
public class MyPostgresQueryInterceptor implements PostgresQueryInterceptor {
  @Override
  public String intercept(String sql, CeMcpTool tool, EngineSession session, Map<String, Object> args) {
    // enforce tenant/schema rules, sql normalization, org conventions, etc.
    return sql;
  }
}
```

## Safety model

1. Metadata table and column identifiers are validated against a strict safe identifier regex.
2. Service only performs table reads from configured knowledge catalogs.
3. No DML/DDL execution path exists in semantic catalog handler.
4. Executable SQL is delegated to separate tool (`postgres.query`) with read-only guardrails.
