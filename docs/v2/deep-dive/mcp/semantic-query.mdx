# MCP Semantic Query (`db.semantic.query`)

This page documents the semantic-query engine under:

`com.github.salilvnair.convengine.engine.mcp.query.semantic`

It is the Text-to-AST-to-SQL runtime for ConvEngine MCP mode `semantic`.

<Highlight type="info" title="Core contract">
Semantic Query generates AST JSON with LLM, compiles SQL deterministically in Java, enforces policy guardrails, executes read-only SQL, and returns auditable result summaries.
</Highlight>

## jOOQ in Semantic Query

jOOQ is not used as a codegen-first ORM here. It is used as a deterministic SQL construction layer controlled by the semantic compiler pipeline.

Flow:
1. LLM generates AST JSON.
2. AST is canonicalized and validated.
3. SQL handlers build clauses with jOOQ DSL + controlled SQL tokens.
4. SQL is emitted with named params.
5. JDBC executes with bind values.

## JGraphT in Semantic Query

JGraphT powers schema traversal and shortest join-path resolution. It prevents LLM from inventing joins by computing paths from real FK metadata.

Flow:
1. Graph engine refreshes vertices/edges from schema inspector FK joins.
2. Join-path resolver chooses base table and required tables from retrieval.
3. Shortest paths are found with hop limits.
4. Resolved edges are de-duplicated and returned to compiler.

### JGraphT traversal examples

<Tabs groupId="semantic-jgrapht-examples">
  <TabItem value="graph1" label="Example 1: Graph build" default>

```java
// JGraphTSchemaGraphEngine.refreshGraph()
for (Map<String, Object> row : joins) {
  String sourceTable = text(row.get("source_table"));
  String targetTable = text(row.get("target_table"));
  nextGraph.addVertex(sourceTable);
  nextGraph.addVertex(targetTable);
  nextGraph.addEdge(sourceTable, targetTable);
  nextGraph.addEdge(targetTable, sourceTable); // reverse edge
}
```

```java
while (!queue.isEmpty()) {
  NodePath current = queue.poll();
  if (current.node().equals(toTable)) {
    return toEdges(current.path());
  }
  for (String next : neighbors(current.node())) {

      queue.add(new NodePath(next, nextPath));
      visited.add(next);

}
```

## AST model capabilities

### Operators in AST model

`AstOperator.java` includes:
- `ILIKE`
- `BETWEEN`
- `IN`
- `NOT_IN`
- `IS_NULL`
- `IS_NOT_NULL`

### New AST nodes

- Boolean tree: `AstFilterGroup.java`
- Exists: `AstExistsBlock.java`
- Subquery filters: `AstSubqueryFilter.java`, `AstSubquerySpec.java`
- Windows: `AstWindowSpec.java`
- Full v1 payload: `SemanticQueryAstV1.java`

### Validation coverage

`DefaultAstValidator.java` validates:
- entity and field existence
- operator and value compatibility (`ILIKE` string-only, `BETWEEN` two values, `IN`/`NOT_IN` array)
- exists blocks
- subquery filters
- window specs
- groupBy/metric constraints

### SQL rendering coverage

- <span className="ce-semantic-op">LIKE</span>/<span className="ce-semantic-op">comparison</span>: <span className="ce-semantic-chip">DefaultComparisonAstOperatorHandler.java</span>
- <span className="ce-semantic-op">BETWEEN</span>: <span className="ce-semantic-chip">DefaultBetweenAstOperatorHandler.java</span>
- <span className="ce-semantic-op">IN</span>/<span className="ce-semantic-op">NOT_IN</span>: <span className="ce-semantic-chip">DefaultInAstOperatorHandler.java</span>
- <span className="ce-semantic-op">IS_NULL</span>/<span className="ce-semantic-op">IS_NOT_NULL</span>: <span className="ce-semantic-chip">DefaultNullCheckAstOperatorHandler.java</span>
- <span className="ce-semantic-op">ROW_NUMBER</span> window: <span className="ce-semantic-chip">DefaultRowNumberWindowFunctionHandler.java</span>

## Request flow by package (quick reference)

1. `runtime/handler`: `DbSemanticQueryToolHandler` receives `db.semantic.query`.
2. `runtime/core + runtime/pipeline + runtime/stage/*`: `SemanticQueryRuntimeService` runs stage DAG.
3. `retrieval/*`: ranks candidate entity/tables.
4. `graph/*`: resolves join edges/path.
5. `llm/*`: generates AST JSON.
6. `ast/*`: canonicalize + normalize + validate.
7. `sql/*`: deterministic compile + policy.
8. `execute/*`: SQL run + row map.
9. `summary/*`: final user-facing summary payload.
10. `model/*`: shared semantic-layer model registry.

join_hints:
  zp_request:
    commonly_joined_with: [zp_disconnect_order, zp_ui_data]

rules:
  allowed_tables: [zp_request, zp_disconnect_order]
  deny_operations: [DELETE, UPDATE, DROP]
  max_result_limit: 500`}
</CodeBlockToggle>

Current status:
- These tags are excellent documentation for model authors.
- They are not currently bound to `SemanticModel` records, so runtime ignores them.
- Effective runtime limit/timezone/dialect come from `application.yml` (`convengine.mcp.db.semantic.*`).
- DML/DDL guardrails are enforced by SQL policy/guardrail steps, not YAML `rules`.

Other descriptive tags in sample:
- `example_questions`, `allowed_values`, `example` are currently prompt/documentation aids and are not strict runtime-enforced model fields.

## AST + SQL impact walkthrough (your metric error example)

User question:
- `show billbank gaps with request and order details`

LLM generated AST (problem shape):

<CodeBlockToggle title="Invalid mixed AST (detail + metric)" language="json" defaultOpen={false}>
{`{
  "entity": "BillingRecord",
  "select": ["billbankId","accountId","disconnectId","recordStatus","pendingBillAmount","updatedAt"],
  "group_by": [],
  "metrics": ["billbank_gap_count"],
  "having": {
    "op": "AND",
    "conditions": [{"field":"billbank_gap_count","op":">","value":0}],
    "groups": []
  }
}`}
</CodeBlockToggle>

Why validator rejects:
- Metrics + detailed select fields requires grouping.
- With empty `group_by`, SQL semantics are invalid.

With current normalizer fix:
- Mixed aggregate/detail AST is demoted to detail query.
- `metrics/having/group_by` are removed deterministically before validation.

<CodeBlockToggle title="Normalized AST after fix (detail mode)" language="json" defaultOpen={false}>
{`{
  "entity": "BillingRecord",
  "select": ["billbankId","accountId","disconnectId","recordStatus","pendingBillAmount","updatedAt"],
  "group_by": [],
  "metrics": [],
  "having": {"op":"AND","conditions":[],"groups":[]}
}`}
</CodeBlockToggle>

Typical SQL shape after fix:

<CodeBlockToggle title="Compiled SQL shape (detail mode)" language="sql" defaultOpen={false}>
{`select
  t0.billbank_id as billbankId,
  t0.account_id as accountId,
  t0.zp_disconnect_id as disconnectId,
  t0.record_status as recordStatus,
  t0.pending_bill_amount as pendingBillAmount,
  t0.updated_at as updatedAt
from zp_billbank_record t0
