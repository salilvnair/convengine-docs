# MCP Example 5 - Semantic Query (Zapper)

This example is the semantic-query runtime equivalent of Example 4.

Difference from Example 4:
- Example 4 uses Database Knowledge Graph (DBKG) playbooks.
- Example 5 uses semantic retrieval + schema graph + LLM AST + deterministic SQL compiler.

<Highlight type="info" title="What this validates">
This example validates your semantic pipeline end-to-end: retrieval, join path, AST generation, AST validation, SQL compile, SQL execute, and markdown-table summary.
</Highlight>

<CodeBlockToggle title="Generated SQL pattern (example)" language="sql" defaultOpen={true}>
{`-- Compiled by DefaultAstQueryAssemblyClauseHandler from canonical AST.
-- Table aliases (t0, t1, t2) assigned in join-path order.
-- WHERE clause built by DefaultComparisonAstOperatorHandler (EQ, GTE).
-- Named params (:p1, :p2) generated by OperatorHandlerContext.nextParam().
-- __limit param injected by compiler for SELECT-only guardrail.
select "t1"."zp_request_id" as "requestId",
       "t1"."account_id" as "accountId",
       "t1"."request_status" as "requestStatus",
       "t1"."requested_at" as "requestedAt",
       "t0"."don" as "disconnectOrderId",
       "t0"."downstream_status" as "downstreamStatus",
       "t0"."zp_disconnect_id" as "disconnectId",
       "t2"."zp_asr_team_notes" as "teamNotes"
from "zp_disconnect_order" as "t0"
left outer join "zp_request" as "t1" on "t0"."zp_request_id" = "t1"."zp_request_id"
left outer join "zp_ui_data" as "t2" on "t1"."zp_request_id" = "t2"."zp_request_id"
where (("t1"."request_status" = :p1) and ("t1"."requested_at" >= :p2))
LIMIT :__limit`}
</CodeBlockToggle>

<CodeBlockToggle title="Compiler classes used" language="text">
{`-- SQL compiler class chain (in execution order):
DefaultSemanticSqlCompiler           -- entry: builds CompileWorkPlan from AST + model
DefaultAstQueryAssemblyClauseHandler -- core: orchestrates SELECT/FROM/JOIN/WHERE/GROUP/HAVING/ORDER/LIMIT
DefaultPredicateClauseHandler        -- WHERE: resolves filter groups recursively
DefaultMetricClauseHandler           -- SELECT + HAVING: injects metric expressions from semantic model
DefaultWindowClauseHandler           -- SELECT: builds OVER() window expressions
DefaultExistsClauseHandler           -- WHERE: builds EXISTS/NOT EXISTS subqueries
DefaultSubqueryClauseHandler         -- WHERE: builds correlated subquery filters
DefaultFunctionClauseHandler         -- SELECT: function wrappers (DATE_TRUNC placeholder)
DefaultComparisonAstOperatorHandler  -- per-filter: EQ, GT, GTE, LT, LTE, LIKE, ILIKE
DefaultBetweenAstOperatorHandler     -- per-filter: BETWEEN
DefaultInAstOperatorHandler          -- per-filter: IN, NOT_IN
DefaultNullCheckAstOperatorHandler   -- per-filter: IS_NULL, IS_NOT_NULL`}
