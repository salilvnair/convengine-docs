# CE Builder Studio

<div className="ce-schema ce-intro-box">
The <strong>CE Builder Studio</strong> is a drag-and-drop canvas for authoring ConvEngine configuration tables. Each block on the canvas maps 1:1 to a <code>ce_*</code> database table row. When you click <strong>Deploy</strong>, the canvas serializes to bulk upsert payloads and posts them to the backend, then triggers a cache refresh so the running engine picks up the new config with zero downtime.
</div>

## Layout

```
┌────────────────┬──────────────────────────────────┬──────────────────┐
│  Block Palette │        ReactFlow Canvas           │  Editor Panel    │
│  (left rail)   │   (drag blocks, wire edges)       │  (right rail)    │
│                │                                   │  SubBlock fields │
├────────────────┴──────────────────────────────────┴──────────────────┤
│  Toolbar: [Deploy ▶]  [Load Existing]  [Clear]   status messages     │
└──────────────────────────────────────────────────────────────────────┘
```

| Zone | Component | Description |
|------|-----------|-------------|
| Left | `CeBlockPalette` | Palette of all available CE block types, organized by category |
| Center | `CeCanvas` | ReactFlow canvas. Drag blocks onto it to add rows. Wire edges between blocks to express relationships. |
| Right | `CeEditorPanel` | Property inspector for the selected block. Renders sub-block fields from the block definition. |
| Bottom toolbar | `CeBuilderPage` | Deploy button, Load Existing dialog, Clear canvas, status/error messages |

## Deploy flow

Clicking **Deploy** triggers `deployCanvas()` from `ceConfig.api.js`:

1. The canvas serializes each block to a flat column-value map (subBlocks → row payload).
2. For each block type group, `POST /api/v1/config/<table>/bulk` is called with an array of row upserts.
3. If all upserts succeed, `POST /api/v1/cache/refresh` is called so the backend's static cache reloads without restart.
4. The toolbar shows per-table success/failure so you can pinpoint which table failed.

<Highlight type="warning" title="Backend endpoints status">
The <code>/api/v1/config/*</code> per-table CRUD endpoints are planned for the Spring Boot backend. Currently ConvEngine rows are authored via SQL Developer / DBeaver and then refreshed via <code>POST /api/v1/cache/refresh</code>. The CE Builder Studio UI transport layer (<code>ceConfig.api.js</code>) is already wired up and ready to connect once the backend endpoints land.
</Highlight>

## Load Existing

The **Load Existing** toolbar button opens `LoadExistingDialog`, which allows importing a previously exported canvas JSON to restore a saved configuration layout.

---

## Intended workflow

1. Open CE Builder Studio from the top nav (label: **Builder Studio**).
2. Drag blocks from the left palette onto the canvas — one block per `ce_*` row you want to create or update.
3. Fill in the sub-block fields in the right Editor Panel.
4. Wire edges between blocks to document relationships (e.g., connect an `intent` block to its `prompt_template` and `rule` blocks for visual clarity — edges are not required for deploy).
5. Click **Deploy** to upsert all blocks to the database and trigger a cache refresh.
6. Test the new config by sending a chat message to the ConvEngine chat panel.
